# Week 1 (W21): May 18–23, 2026 — The Setup

A Berlin dialect translator gets its tooling, its plumbing, and its first real feature — all in five days.

---

## Narrative

The week opens on May 18 with a single long session of reading. Not writing, not building — just reading. Claude moves through the berliner-schnauze codebase from top to bottom: stores, components, composables, pages, configuration. Twenty-seven distinct observations logged in under seven minutes of wall time. This is reconnaissance. The project is already built — a static Astro site backed by a WordPress GraphQL API, Vue 3 islands for interactivity, Nanostores for client state, Orama for in-browser search, Cloudflare Pages for deployment. Claude is learning the shape of something that Felix has already made.

Then three days pass. Nothing logged. The calendar jumps to May 21.

What follows on May 21 is dense and sequential: the kind of day where one thing leads directly into the next. It starts with automation infrastructure. The Claude Code harness gets hooks configured for linting and `.env` file protection. CLAUDE.md gets audited — import aliases corrected, missing scripts documented, a wrong store name fixed. The project's self-documentation is brought into alignment with its actual configuration.

From there, the day escalates into dependency work. GraphQL codegen jumps from v5 to v6/v7 — a major migration. The generated type file swells past TypeScript's file size limit. Enum imports break because the new codegen emits union types instead of object constants. A chain of sessions works through each failure. By the end, codegen runs clean and lint passes.

Then a different kind of breakage surfaces: the word-of-the-day REST endpoint is returning 401. What looks like a frontend authentication problem turns out to be a WordPress filter priority issue — a global JWT authentication filter is blocking a route that was intended to be public. After several sessions probing the WordPress side, the filter is fixed at the correct priority and the endpoint goes green. The frontend auth headers and token imports that had accumulated in the store are cleaned up.

The evening of May 21 brings the week's main feature: a PWA cache inspection UI. Felix wants a page where users can see what's stored in the service worker cache. Claude designs it, then implements it in phases. A `CacheBucket` type and helper functions first, then a `useCacheStorage` composable with reactive state and `loadCaches`. Each task verified with tests before the next begins. A `ConfirmDialog` component arrives to replace native browser prompts. Offline status detection is fixed — the original implementation checked `navigator.onLine`, which is unreliable on page reload; the fix performs an actual fetch probe.

May 22 runs from midnight into the small hours: Service Worker status indicators get Lucide icons instead of emoji. TypeScript path aliases get corrected in `tsconfig.json`. A Vue expert review of the new composable surfaces four issues — a `computed()` wrapper around a static capability check, lifecycle hooks registered in the wrong order, a date reference equality bug, and a performance bottleneck from sequential blob reads. Three are fixed immediately.

May 23 is cleanup and consolidation. Twenty-five TypeScript errors from `pnpm astro check` are resolved in a single focused session — null narrowing, wrong GraphQL import paths, incorrect prop shapes. SCSS linting gets configured and surfaced in the VS Code IDE. Then, in the final hour of the week, Infisical secret management gets integrated: `.infisical.json`, wrapped dev scripts, OIDC in the release pipeline. CLAUDE.md is updated to document it. The week ends with a note about five code review findings in the Infisical integration — left open, not yet resolved.

Next week inherits a project that runs and lints cleanly, but with loose threads in secrets management, image types, and PWA performance.
