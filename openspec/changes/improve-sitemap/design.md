## Context

The sitemap is generated by the `@astrojs/sitemap` integration in
`astro.config.mjs`. Today it does two things:

- `filter`: drops `/settings*` and the BON `/share` route.
- `serialize`: for URLs matching `/wort/<slug>`, looks up a `lastmod` from
  `getWordDates()` (`src/services/queries/getSitemapWordDates.ts`) — a paginated
  WPGraphQL fetch of `berlinerWords { slug modifiedGmt }`, cached in a
  module-level promise for the build.

All other routes (magazine articles, Themen, changelog, legal/static pages) are
emitted with no `lastmod`. Magazine articles *are* emitted (they're SSG'd via
`getStaticPaths` in `src/pages/magazin/[postSlug].astro`), but this has never been
asserted and the user is unsure. `getSitemapWordDates.ts` deliberately reads
`process.env.WP_API` directly and uses `fetch` (not urql) so it is safe to import
from `astro.config.mjs`, which runs outside the Astro/urql runtime.

**Backend check (done):** `posts` expose both `date` and `modifiedGmt`.
`BerlinerischThema` is a **taxonomy term** (`isTermNode`), not a content node — it
has no `date`/`modifiedGmt` field at all. Decision (confirmed with the user):
Themen pages stay in the sitemap **without** a `lastmod`; only words and posts get
dates. This narrows scope to two content types.

## Goals / Non-Goals

**Goals:**

- `lastmod` on `/wort/` and `/magazin/` URLs.
- A build-time assertion (test) that every published magazine article is present
  and that magazine/themen routes aren't filtered out.
- One backend query per dated content type per build, cached — no per-URL fetching.

**Non-Goals:**

- `lastmod` on `/themen/` URLs (taxonomy terms have no modification date; deriving
  one from member words is out of scope for this change).
- Splitting the sitemap into per-type files (auto-split at 45 000 URLs already
  exists; corpus is far below that).
- `changefreq` / `priority` hints (Google ignores them).
- Switching away from `@astrojs/sitemap` or adding any dependency.

## Decisions

### Decision: Generalize `getWordDates()` into a slug→lastmod lookup per content type

Extend `getSitemapWordDates.ts` to expose one cached fetcher per dated content
type — `getWordDates()` and `getPostDates()` — each returning a
`Map<slug, isoDate>`, both following the existing paginated-fetch + module-cache
pattern. The `serialize` function matches the URL against `/wort/` and `/magazin/`
and pulls `lastmod` from the matching map.

- **Why**: reuses the proven, urql-free, env-var-driven fetch pattern that is
  already safe to import into `astro.config.mjs`. Keeps a single per-build cache
  per type.
- **Alternative considered**: reuse the existing urql-based `fetchAllPosts()` /
  `getThemen()` services. Rejected — those import the urql client and Astro env
  (`astro:env`), which is not safely importable from `astro.config.mjs` (runs
  before the Astro runtime). The GraphQL queries there also don't currently select
  `modifiedGmt`.
- **Alternative considered**: one combined query for all types. Rejected — three
  separate custom-post-type root fields; a single query buys nothing and couples
  the fetchers.

### Decision: Derive `lastmod` from `modifiedGmt`, not `date`

Use the backend's modified timestamp (append `Z`, `toISOString()`) exactly as the
word path already does. The build-time post query selects `slug` + `modifiedGmt`.

- **Why**: `lastmod` semantically means last modification, not publish date.

### Decision: Do not split the sitemap

Keep the default single-index output. `@astrojs/sitemap` splits automatically at
`entryLimit` (45 000). Record 45 000 as the revisit threshold.

- **Why**: current corpus (hundreds–low thousands of URLs) is orders of magnitude
  below the limit; manual splitting adds config and cognitive load for zero SEO
  benefit at this scale.

### Decision: Lock in magazine inclusion with a test

Add a unit test asserting the generalized date service returns post entries, and
(cheaply) that the `filter` predicate keeps `/magazin/` and `/themen/` while
dropping `/settings` and `/share`. The `filter` is a pure function — extract it so
it can be imported and tested without booting a build.

- **Why**: the user's core doubt is "do magazine articles even show up?" — encode
  the answer as a regression guard rather than a one-off manual check.

## Risks / Trade-offs

- **Extra build-time query (1 more paginated fetch for posts)** → Mitigation:
  cached per build, runs alongside the existing word fetch; post count is small.
- **Backend auth in `astro.config.mjs`** → the file already reads
  `WP_AUTH_USER/PASS` for the word fetch; the new fetcher reuses the same headers.
  No new secret surface.

## Migration Plan

Pure build-time change, no runtime/data migration. Deploy = next Cloudflare Pages
build regenerates the sitemap. Rollback = revert the `astro.config.mjs` +
service diff; the sitemap returns to word-only `lastmod`.

## Open Questions

- None. Backend field availability confirmed: posts expose `modifiedGmt`; Themen
  are taxonomy terms with no date (excluded from `lastmod` by decision above).
